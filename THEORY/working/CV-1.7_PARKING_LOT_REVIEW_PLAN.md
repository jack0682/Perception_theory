## 3. Plan stages

### Stage 0 — Inventory (1 day)

**Output:** `THEORY/working/CV-1.7_parking_lot_inventory.md` (new file)

Steps:
1. Run a script that lists all `THEORY/working/{MF,SF,E,CE,C}/*.md` files modified in `[2026-04-30, 2026-05-01]` that lack a "CV-1.7 candidate" header AND do not appear in CHANGELOG.md as "promoted" or "retired".
2. For each file, extract: line count, last commit, single-sentence purpose (from the file's H1 title or first paragraph), claimed Cat status (A/B/C/conjecture/working).
3. Group by cluster (σ_rich, σ-fingerprint, K-Selection, reconciliation, commitment packets, auxiliary).
4. Identify any cross-references that would break if the file were retired.
5. The inventory file is the audit-ready checklist; subsequent stages fill in critic verdicts per file.

**Acceptance criterion:** 17 (or actual count) files enumerated with all metadata fields populated. No file in the W5 Wave 3 timestamp window left out.

### Stage 1 — Per-file Cat-status self-assessment (2-3 days, batch)

**Output:** Update each parking-lot file's header to include:
```
> Status: CV-1.7 parking lot (added <date>); current self-classification: <Cat A absolute / Cat A conditional / Cat B target / Cat C target / Working only / Conjecture>; pending critic verdict.
> Cluster: <σ-rich foundation / σ-fingerprint / K-Selection / reconciliation / commitment / auxiliary>.
> Promotion path (if any): <e.g. "Commitment 18 candidate at CV-1.7", or "T-σ-Theorem-4 γ-path input">.
> Last reviewed: 2026-05-04 (audit pass; self-assessment only — no external critic).
```

This step is mechanical and can be done in one sitting. It surfaces the file metadata and makes the parking-lot status visible at-a-glance.

**Acceptance criterion:** all parking-lot files have the header.

### Stage 2 — Critic dispatch by cluster (5-10 days, sequential)

Critic dispatches happen one cluster at a time, in priority order. Each cluster produces a verdict file recorded in `THEORY/CHANGELOG.md`.

Priority order:

1. **Reconciliation drafts (2 files / 760 lines, ~1 day).** Smallest cluster, directly feeds the T-σ-Theorem-4 γ/β/α audit. Critic verdict: "are the discrete-grid extrapolation arguments sound?" Outcome: either feeds γ-path audit input or is retired.
2. **σ-rich foundation (8 files / 2,764 lines, ~3-4 days).** Largest cluster, central to the OP-0008 σ^A K-jump non-determinism path B. Critic verdict per file: "is the augmentation step (centroid / orientation / Wigner / Φ_rich composition) mathematically sound?" Outcome: each file gets Cat B/C or retired.
3. **K-Selection (5 files / 1,915 lines, ~2 days).** Composite of free-energy, Kramers, numerical anchor, and Commitment 16 lineage. Critic verdict: "are the four sub-files compatible (no contradiction)? are the (V1)-(V7) numerical gates well-defined?" Outcome: Commitment 19 axiom packet either becomes ready for CV-1.7 promotion or is restructured.
4. **Commitment packets (2 files / 835 lines, ~1 day).** Pre-promotion drafts of Commitment 18 (σ-rich) and Commitment 19 (K-Selection). Critic verdict: "is the axiom statement precise? is the canonical-conjugation rule respected?"
5. **σ-fingerprint (2 files / 539 lines, ~1 day).** NQ-264 R23 candidate. Lower priority because it is an empirical anchor task.
6. **Auxiliary (3 files / 857 lines, ~1 day).** Categorical / π_1 / Lie algebra explorations. Critic verdict: "is this a load-bearing scaffolding for a canonical promotion candidate, or speculation that should sit in a separate exploratory subdirectory?"

Total estimate: ~10 working days, sequential to keep critic context coherent. Could be parallelised across user-dispatched native teams if desired, but sequential is safer given the inter-cluster dependencies (σ-rich and K-Selection both feed Commitment packets).

**Acceptance criterion:** every parking-lot file has a critic verdict (ACCEPT / REVISE / RETIRE), recorded in CHANGELOG.

### Stage 3 — Disposition (1-2 days)

For each file with a verdict:

- **ACCEPT** → file remains in `working/<area>/`; promotion path is documented in its header; CV-1.7 release plan can pick it up.
- **REVISE** → file remains; revision tasks are listed in its header; subsequent revision triggers another critic dispatch.
- **RETIRE** → file is moved to `_archive/cv17_parking_retired_2026-05-XX/<name>.md` with a brief retirement note. Retirements are recorded in CHANGELOG.

**Acceptance criterion:** parking lot is empty (every file is either accepted into the CV-1.7 release plan, sitting in revision, or archived).

## 4. Risks and mitigations

- **Critic fatigue.** 17,269 lines (Stage 0 audited count) is a lot. Mitigation: cluster-by-cluster dispatch with per-cluster verdict windows; the user can pause between clusters. *(Original W5 narrative estimate "8,145 lines" was ~2.1× under-counted.)*
- **Inter-cluster dependency.** σ-rich and K-Selection both feed Commitment packets. Mitigation: do the dependents (σ-rich, K-Selection) before the depending (Commitment packets).
- **Re-spawn risk.** A critic verdict of REVISE may spawn new working files, growing the parking lot. Mitigation: Stage 3 disposition explicitly counts new working files against the parking-lot total — there should be a downward trajectory.
- **Silent abandonment recurrence.** This is what the deleted W6 plan did. Mitigation: each cluster's critic verdict is recorded in CHANGELOG with date; the parking-lot inventory file from Stage 0 is the persistent ledger.

## 5. Decision points for user

The following decisions are needed before Stage 0 begins:

1. **Execution mode.** Sequential (one cluster at a time, lead handles), or parallel (native teams handle multiple clusters concurrently).
2. **Critic source.** Lead-side critic dispatches (cheap, fast), or external native-team critic (slower, higher independence).
3. **Disposition policy for borderline files.** Default REVISE or default RETIRE? Conservative (REVISE) preserves work but grows the queue; aggressive (RETIRE) shrinks the queue but loses material.
4. **Cluster ordering.** Reconciliation drafts first (feeds γ-path, time-sensitive), or σ-rich first (largest cluster, biggest unknown).

The plan as drafted assumes sequential / lead-side critic / default-REVISE / reconciliation-first. Any of these can flip with a one-line user decision.
